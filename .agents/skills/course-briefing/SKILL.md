# 说课

## 概述

当用户需要“说课稿、课程设计汇报、课程评审说明、讲师试讲前说明、课程开发成果汇报、课程方案讲解口径”时，使用这个能力。

说课不是把课程内容重新讲一遍，也不是写招生介绍。它的目标是向评审者或相关方说明：这门课为什么要开、为谁解决什么问题、目标如何设定、结构为什么这样搭、内容为什么这样取舍、教学过程如何促进学习、评估如何验证目标达成。

说课必须基于已有课程开发产物，不得脱离课题分析、课程目标、课程结构、内容组织和过程设计进行泛泛包装。

## 输入优先级

优先使用以下输入：

1. `$topic-analysis` 的课题判断、业务背景、学员画像、工作场景、绩效差距和 KAS 内容地图。
2. `$course-objectives` 的课程目标、目标类别/水平和评估证据。
3. `$course-structure` 的模块结构、主导结构逻辑和模块承接关系。
4. `$content-organization` 的内容取舍、内容分类、案例/练习素材和工作应用。
5. `$process-design` 的教学流程、开场导入、教学活动、时间分配、讲师/学员动作和学习衡量。
6. `$quiz-generation` 的考试题、随堂测试或学习评估方案，如有。
7. 用户给出的说课对象、时长、场景和格式要求。

如果缺少核心课程设计产物，先判断能否基于当前材料生成“临时说课稿”。正式评审型说课至少需要课程主题、课程目标、课程结构和教学过程；缺失时应列出需要补充的材料。

## 说课对象

先判断说课听众是谁，再调整重点：

- `业务负责人`：强调业务问题、绩效差距、课程收益和转化到工作场景的价值。
- `培训管理者`：强调课程目标、结构逻辑、教学方法、评估方式和可复制交付。
- `评审专家`：强调课程开发依据、目标可衡量性、内容取舍、教学活动和评估闭环。
- `讲师团队`：强调授课主线、关键活动、讲师动作、时间节奏和风险控制。
- `学员代表`：强调学习价值、参与方式、课程产出和回岗应用。

不要所有对象都用同一套说法。

## 核心流程

### 1. 明确说课场景

先确认或推断：

- 说课对象是谁。
- 说课总时长是多少。
- 说课用于评审、汇报、试讲前说明、课程推广，还是讲师共备。
- 是否需要完整讲稿、结构提纲、逐页讲解口径或评审答辩准备。

如果用户未指定时长，默认给 5-8 分钟口头说课稿；若课程复杂，可额外提供 3 分钟精简版和 10 分钟完整版建议。

### 2. 建立说课主线

默认使用这条主线：

```text
为什么开这门课 -> 学员要达成什么目标 -> 课程如何搭结构 -> 内容如何支撑目标 -> 教学过程如何促成学习 -> 如何评估效果 -> 课程价值与落地保障
```

这条主线必须体现课程开发逻辑，而不是平铺课程目录。

### 3. 提炼课程设计依据

从前置产物中提炼：

- `业务依据`：课程服务哪个业务目标、质量问题、项目问题或绩效问题。
- `学员依据`：目标学员是谁，他们在工作中遇到什么痛点。
- `目标依据`：课程目标如何从绩效差距转化而来。
- `结构依据`：为什么采用当前模块顺序和结构逻辑。
- `内容依据`：哪些内容必讲、必练、下沉或排除，原因是什么。
- `过程依据`：为什么选用这些教学方法、案例、练习和反馈方式。
- `评估依据`：如何确认学员学会、会用、愿意用。

### 4. 组织说课内容

正式说课建议包含 7 个部分：

1. `课程背景`：业务问题、学员场景和课程开发原因。
2. `课程定位`：课程对象、课程边界、解决的问题和不解决的问题。
3. `课程目标`：知识、技能、态度或核心能力目标，以及可观察证据。
4. `课程结构`：模块主线、模块关系和学习路径。
5. `内容组织`：核心内容、案例/练习、内容取舍和工作应用。
6. `教学过程`：开场导入、关键活动、练习反馈、时间节奏和讲师引导。
7. `学习评估`：课堂衡量、考试/测验、行动计划或回岗应用评估。

如果说课时间很短，保留“背景痛点 -> 目标 -> 结构 -> 关键教学设计 -> 评估闭环”五段即可。

### 5. 写成可讲的话

说课稿要像面对评审或负责人自然说明，而不是论文摘要。要求：

- 语言简洁、口语化、专业。
- 每段都有明确功能。
- 不堆模型名，不空泛夸课程价值。
- 用“因此、所以、为了”串起设计逻辑。
- 关键处可标注 `[停顿]`、`[指向课程结构图]`、`[举例]`、`[过渡]`。

避免：

- 只介绍课程目录。
- 只说“采用案例教学、小组讨论”等空话。
- 只说课程很重要，不说明业务痛点。
- 夸大培训能解决的非培训问题。
- 说课稿与实际课程目标、结构、过程不一致。

### 6. 准备答辩追问

若用于评审或汇报，补充可能被问到的问题：

- 为什么选择这个课题？
- 目标为什么这样写，如何衡量？
- 为什么这样分模块？
- 哪些内容被删掉了，为什么？
- 学员怎么参与，不只是听讲？
- 如何判断培训有效？
- 课程落地最大的风险是什么，如何应对？

答辩回答必须回到课题分析、目标、结构、内容、过程和评估证据。

## 输出格式

除非用户要求其他格式，按以下结构输出：

1. `说课定位`：对象、场景、时长、说课重点。
2. `说课主线`：一句话说明本次说课的逻辑。
3. `说课结构提纲`：按背景、目标、结构、内容、过程、评估、价值列出要点。
4. `完整说课稿`：可直接口头讲述，含必要动作/停顿/过渡提示。
5. `设计亮点`：课程设计中最值得评审关注的 3-5 个亮点。
6. `可能追问与回答`：用于评审答辩或讲师共备。
7. `材料缺口/优化建议`：只列影响说课可信度的关键缺口。

快速请求时，可只输出“说课结构提纲 + 完整说课稿”。

## 质量规则

- 说课必须基于课程开发产物，不凭空包装。
- 每个说法都要能追溯到业务问题、学员场景、课程目标、模块内容或教学过程。
- 不把课程目录当说课主线。
- 不夸大培训效果；对非培训因素要保持边界。
- 必须说明目标如何达成、过程如何支撑、评估如何验证。
- 说课稿要符合指定时长，不能明显超时。
- 面向不同听众调整表达重点。
