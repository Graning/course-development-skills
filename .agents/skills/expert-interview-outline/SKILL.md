# 专家访谈提纲设计

## 概述

当课程开发需要从业务专家、岗位高手、流程负责人或一线骨干那里萃取经验、案例、难点、方法和素材时，使用这个能力。

专家访谈的目标不是让专家泛泛介绍知识，而是把专家在真实工作中的事件、判断、操作步骤、难点处理和成功经验提取出来，转化为课程内容、案例、练习和工具表。

必须严格遵循：

`课题分析定痛点 -> 课程目标定方向 -> 课程结构定框架 -> 访谈提纲定聚焦`

访谈提纲中的所有问题都必须能追溯到输入材料中的业务痛点、课程目标或课程模块；不能新增脱离课程需求的通用问题。

本能力基于附件中的“专家访谈 7 步”：

`说明目的 -> 分享事件 -> 提炼经验 -> 深挖难点 -> 提出方法 -> 收集例子 -> 表达感谢`

## 使用场景

适用于：

- 为新课程开发访谈业务专家。
- 为 `$content-organization` 补充实践经验、案例和工作启发。
- 为 `$process-design` 准备案例、练习任务和讲师点评素材。
- 把专家经验从“他说得懂”转成“学员能学、能练、能用”的内容。
- 访谈前需要准备提纲、话术、追问路径和记录模板。

## 核心流程

### 1. 校验前置输入

生成正式访谈提纲前，必须先检查三类核心输入是否齐全：

- `最终版课题分析表`：应包含业务痛点、学员对象、工作场景、学员“不懂/不会/不愿”等核心难点。
- `最终版课程目标`：应是符合标准格式的完整目标，而不是泛泛培训意图。
- `最终版课程结构`：应至少包含课程模块、关键内容、重难点；如用户声明“四级细分”，则按四级结构对齐。

如果缺少以上任一核心输入，不生成完整访谈提纲；只输出“缺失输入清单”和“需要补充的材料”。访谈对象、访谈时长、访谈形式可以作为补充输入，缺失时可做假设并标注。

### 2. 建立对齐关系

在设计问题前，先建立三张映射：

- `痛点映射`：课题分析中的不懂/不会/不愿 -> 需要深挖的问题。
- `目标映射`：每条课程目标 -> 需要萃取的经验、案例、方法或标准。
- `模块映射`：每个核心教学模块 -> 需要访谈确认或补充的内容。

没有对应业务痛点、课程目标或课程模块的问题，一律删除。课程结构中的每个核心模块都必须有对应访谈问题；每条课程目标都必须有对应经验萃取问题；“不懂/不会/不愿”三类难点都必须有深挖问题。

### 3. 按 7 步搭建访谈提纲

默认使用 7 步结构：

1. `说明目的`：让专家知道为什么访谈、访谈多长、如何使用结果。
2. `分享事件`：引导专家讲一个真实、典型、与课题相关的工作事件。
3. `提炼经验`：让专家确认处理步骤、判断依据、关键动作和成功因素。
4. `深挖难点`：追问事件中的困难、障碍、重复点、强调点、转折点。
5. `提出方法`：萃取专家对难点的解决方案、操作原则、注意事项。
6. `收集例子`：补充典型案例、正反例、常见误区和适合课堂使用的素材。
7. `表达感谢`：确认内容、约定后续校对方式，并说明素材应用方式。

每一步的问题都必须贴合本次课程的核心需求，不得直接套用通用话术。详细话术、难点和解决建议见 [interview-seven-steps.md](references/interview-seven-steps.md)。

### 4. 区分核心必问和拓展选问

所有问题必须分为：

- `核心必问问题`：必须 100% 覆盖课程重难点、核心痛点、课程目标和核心模块；确保 1 小时访谈能拿到课程必备素材。
- `拓展选问问题`：用于时间充足时补充边界、变体、反例、工具、模板和更多案例。

核心必问问题优先放在前面，并控制数量。不要为了完整而堆砌问题。

### 5. 为核心问题配置 STAR 追问

核心问题必须搭配 STAR 深挖话术：

- `S - Situation 情境`：当时发生在什么场景？背景、角色、约束是什么？
- `T - Task 任务`：当时您要完成什么任务？目标、标准、风险是什么？
- `A - Action 行动`：您具体做了什么？先后步骤、判断依据、工具方法是什么？
- `R - Result 结果`：结果如何？怎么验证有效？有什么经验、教训或可复用方法？

STAR 追问要围绕输入材料中的痛点、目标和模块，不要变成泛泛访谈模板。

### 6. 设计访谈话术

话术应自然、尊重、具体：

- 开场说明要让专家放松，并知道访谈不会考核他。
- 分享事件时要给专家足够空间。
- 追问难点时要避免评价专家做得对不对。
- 总结确认时要复述专家观点，让专家修正或补充。
- 收尾时要说明后续可能需要专家确认内容。

使用“复述 + 追问”：

`您刚才提到……，我理解这是关键点。能否再展开一下，当时您具体怎么判断/怎么做？`

### 7. 设计结果整理方式

访谈提纲必须考虑后续如何整理结果。建议输出字段包括：

- 典型事件。
- 工作场景。
- 关键任务。
- 关键动作。
- 判断依据。
- 难点/障碍。
- 解决方法。
- 常见错误。
- 可转化为课程的内容。
- 可转化为案例/练习/工具表的素材。
- 需要专家二次确认的问题。

## 输出格式

除非用户要求其他格式，按以下结构输出：

1. 输入校验：说明是否已收到最终版课题分析表、课程目标、课程结构；缺失则停止生成正式提纲。
2. 对齐关系：简要列出痛点、课程目标、课程模块与访谈问题的映射逻辑。
3. 访谈提纲表：严格使用 Markdown 表格输出，可复制到 Excel，包含 6 列：
   `访谈环节 | 对应7步法步骤 | 对应课程模块/课程目标/业务痛点 | 访谈问题 | STAR追问话术 | 记录栏`
4. 问题标注：每个问题必须标明 `核心必问` 或 `拓展选问`。
5. 访谈结果转化建议：说明萃取内容将进入课程目标、结构、内容组织或过程设计的哪个位置。

快速请求时，也必须保留 6 列访谈提纲表；可以压缩输入校验和转化建议。

## 质量规则

- 每个问题都要服务课程开发目的。
- 所有问题都必须可追溯到课题分析痛点、课程目标或课程结构模块。
- 无对应业务痛点、无对应课程目标、无对应模块的问题一律删除。
- “不懂/不会/不愿”三类核心难点必须都有对应深挖问题。
- 每条课程目标都必须有对应经验萃取问题。
- 每个核心教学模块都必须有对应访谈问题。
- 先问真实事件，再提炼方法，不要一开始就问抽象经验。
- 核心必问问题必须配 STAR 追问。
- 每个关键经验都要追问“当时怎么判断、怎么做、依据是什么”。
- 难点问题要追问重复点、强调点、转折点和障碍点。
- 专家说得太泛时，用具体场景、具体事件、具体动作拉回。
- 专家说不停时，用复述确认和时间提醒收束。
- 最终结果必须能转化为课程内容、案例、练习或工具表。

## 与课程开发 Skill 的关系

`$expert-interview-outline` 是课程开发链路的支持能力，可在以下阶段使用：

- `$topic-analysis`：补充真实工作场景和绩效差距。
- `$course-objectives`：确认学员需要达成的真实行为。
- `$course-structure`：确认模块主线和关键任务。
- `$content-organization`：萃取实践经验、案例、方法和工具。
- `$process-design`：准备案例、练习、引导问题和点评素材。
