# 结构搭建

## 概述

当用户需要搭建课程大纲、模块结构、章节层级、内容顺序或重构零散材料时，使用这个能力。目标不是写完整教案，而是先把课程的“骨架”搭稳：为什么这样分模块、模块之间是什么逻辑、每个模块承载什么学习结果。

结构必须服务于学员的工作任务和课程目标。不要按资料来源、PPT 页、制度章节或讲师习惯直接堆内容。

## 输入优先级

优先使用这些输入：

1. `$topic-analysis` 的课题边界、学员场景、KAS 内容地图。
2. `$course-objectives` 的课程目标。
3. 用户给出的工作任务、流程、案例、SOP、模板、制度或素材清单。
4. 讲师已有大纲或 PPT 目录。

如果课题边界、学员对象或课程目标缺失，先做必要假设并标注；如果缺口会明显影响结构判断，建议先使用 `$topic-analysis` 或 `$course-objectives`。

## 核心流程

### 1. 明确结构锚点

先确定课程结构服务什么：

- `目标学员`：谁要学，承担什么工作任务。
- `课程目标`：学完后要能完成什么行为或输出。
- `核心场景`：课程服务哪个真实工作场景。
- `内容边界`：哪些必须覆盖，哪些可选，哪些排除。
- `课程时长`：时长决定结构颗粒度和练习深度。

不要在目标不清楚时直接列模块。

### 2. 选择主导结构逻辑

一个课程应有一个主导结构逻辑，必要时局部嵌套另一种结构。

- `流程型`：按工作步骤、项目阶段、操作顺序展开。适合流程执行、项目管理、问题处理、工具操作。
- `要素型`：按并列维度、能力要素、评审标准、关键要素展开。适合模型、标准、能力、清单类课程。
- `WWH`：按为什么、是什么、怎么做展开。适合理念导入、方法推广、产品价值、管理议题。
- `案例问题型`：围绕一个典型问题或案例推进。适合质量问题、项目复盘、复杂判断、经验萃取。
- `任务练习型`：围绕多个递进任务或练习产出推进。适合技能训练、模板填写、评审判断、沟通演练。

如果一个课题包含多个互不相关的逻辑，不要强行合并；建议拆成不同课程或模块。

详细选择规则和结构模板见 [structure-patterns.md](references/structure-patterns.md)。

### 3. 整理和分组内容

把内容先转成“学员需要完成的任务、判断、输出或标准”，再分组：

- 合并重复内容。
- 删除不服务目标的背景内容。
- 把资料标题改写成学习任务或能力标题。
- 区分主线内容、支撑知识、练习材料和附录材料。
- 保持同级模块颗粒度一致。

模块不是资料夹。每个模块都应回答：“这个模块帮助学员完成什么工作动作？”

### 4. 搭建模块层级

默认结构层级：

`课程主题 -> 一级模块 -> 二级要点 -> 关键活动/案例/产出`

一级模块通常 3-5 个；二级要点通常每个模块 2-4 个。若超过这个范围，优先合并、拆分或下沉到附录。

模块标题规则：

- 一级标题表达学习任务、工作阶段、能力要素或问题链路。
- 二级标题表达该模块下的关键判断、方法、步骤、标准或输出。
- 同级标题使用同一种语法和抽象层级。
- 不用“案例分享”“互动讨论”“制度介绍”作为内容标题，除非它本身就是教学活动而非模块。

### 5. 排列顺序和承接关系

检查模块顺序是否符合学习和工作逻辑：

- 从任务前提到任务执行。
- 从标准认知到案例判断。
- 从简单场景到复杂场景。
- 从单点技能到综合应用。
- 从问题识别到原因分析，再到行动闭环。

为每个模块写一句承接说明：为什么这个模块放在这里，它如何连接前后模块。

### 6. 校准结构完整性

结构完成后，逐项校验：

- 每个课程目标都有模块承载。
- 每个模块都能追溯到目标、场景或绩效差距。
- 模块之间没有明显重复。
- 必须内容在主线中，可选内容被压缩或放入附录。
- 标题不是口号、资料名或活动名。
- 同级模块颗粒度一致。
- 结构能支撑后续教学活动设计。

## 输出格式

除非用户要求其他格式，按以下结构输出：

1. 结构判断：推荐使用的主导结构逻辑及理由。
2. 课程结构总览：课程主题、目标学员、结构主线。
3. 模块结构表：一级模块、二级要点、对应目标、关键内容、建议活动/产出。
4. 模块承接逻辑：每个模块为什么放在该位置。
5. 内容取舍：必须覆盖、可选下沉、建议排除。
6. 结构校验：目标覆盖、层级一致性、重复风险、缺口。
7. 待确认问题：只列会影响结构的关键输入。

对于快速请求，可以只输出“推荐结构 + 模块表 + 取舍说明”。

## 质量规则

- 先确定结构逻辑，再命名模块。
- 模块标题必须服务学员任务和课程目标。
- 不按资料目录照搬结构。
- 不把教学方法当作课程内容模块。
- 不把宽泛口号作为一级模块标题。
- 必须内容优先，附加背景后置。
- 同级模块要同一颗粒度。
- 如果结构过大，主动建议拆课或拆模块。

## 与其他能力的关系

使用 `$topic-analysis` 先判断课题边界、业务目标、学员场景和内容范围。

使用 `$course-objectives` 先明确可衡量课程目标。

使用 `$course-structure` 搭建模块、层级和内容顺序。

结构稳定后，使用 `$content-organization` 组织内容；内容稳定后，使用 `$process-design` 设计教学流程、活动、案例、引导问题和验证方案。
